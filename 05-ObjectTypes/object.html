Object Types

In TypeScript, we can create annotations to describe the shape of the object. Accessing a property on that object that isn't defined or performing operations without keeping types in mind will throw errors, not just fail silently.

These object annotations can be passed into functions, or to variables as well.

Can use the object annotations to describe the return type of a function too.

Excess Properties

In JS, if you pass in extra parameters to a function, it doesn't have a problem with that and sends the function on its merry way. Similarly, we can pass in extra properties of an object into a function that makes use of some of the properties and the extra ones will just be ignored.

TypeScript though doesn't like this one bit. When the object is defined in-line of the function call, it wants the properties to line up perfectly.

This doesn't apply if you pass in a variable which is an object that has excess properties, TS is just fine with that, just like JS would be.

Type Alias

Instead of writing out object types in an annotation, we can declare them seperately in a type alias, which is simply the desired shape of the object. This allows us to make our code more readable, and even reuse types elsewhere in our code.

Commonly used even in simple objects, and especially useful for typing nested objects.

Optional Properties

Normally, TS will yell at us if we are missing any of the keys from an object annotation we said would be there. That is what we want it to do.

But keys can also be optional, so TS will chill the fuck out. This is done with a ? before the type annotation. CTD.

readonly Keyword

The readonly keyword can go on objects or arrays and will flag TS to yell at us if we attempt to change those properties / values. If the readonly key was an array or an object, we could add to that but we could not reassign it.

Intersection Types

You can combine two declared Types with an & character.
